Conceptos base
¿Qué es una BAPI y por qué existe?
Antes de ejecutar nada, entendemos el concepto. Una BAPI es la base de casi todo desarrollo de integración en SAP.
SAP Standard Módulo PP Sin código requerido
Una BAPI es como una ventanilla de banco. Tú no entras a la sala de servidores a mover el dinero tú mismo — le das los datos a la ventanilla, ella valida, procesa y te devuelve la respuesta. SAP ya programó esa ventanilla. Nosotros solo la usamos.
— Analogía para equipos funcionales
Definición técnica

BAPI (Business Application Programming Interface) es un módulo de función SAP estándar, con una interfaz estable, que permite crear, modificar o consultar objetos de negocio (órdenes, materiales, proveedores, etc.) de forma programática y segura.

💡
A diferencia de otros módulos de función, las BAPIs son estables entre versiones de SAP. Si un programa las usa hoy, seguirá funcionando en una versión futura del sistema.

Entender la estructura de la BAPI
Antes de escribir una sola línea de código, exploramos la BAPI visualmente. Lo que ves en SE37 es exactamente lo que usará el programa.
SE37 BAPI_PRODORD_CREATE
Cómo acceder
SAP 1. Ir a transacción SE37
2. Buscar: BAPI_PRODORD_CREATE
3. Ejecutar con F8
Las tres pestañas que debes conocer

Lo que nosotros metemos a la función. Son los datos de entrada que necesita para ejecutar su lógica.

📥
Aquí van el material, la planta, el tipo de orden, la cantidad, las fechas. Son los parámetros obligatorios para que la BAPI sepa qué crear.

Lo que la función nos devuelve después de ejecutar. El resultado del proceso.

📤
El campo más importante aquí es ORDER_NUMBER — si la creación fue exitosa, aquí viene el número de la orden recién generada en SAP.

La tabla RETURN es la más importante. Contiene todos los mensajes: éxitos, errores y advertencias.

⚠️
Regla de oro: siempre revisa la tabla RETURN después de ejecutar. Si no hay errores aquí, la operación fue exitosa.

Buenas prácticas
Consultar antes de crear
Un paso que la mayoría salta y que evita el 80% de los errores. Antes de crear, consultamos una orden existente para ver exactamente cómo vienen los datos.
SE37 BAPI_PRODORD_GETDETAIL Lectura · Sin cambios
Por qué usar GetDetail primero

Los campos en SAP tienen formatos específicos — un material puede tener ceros a la izquierda, una planta puede ir con o sin punto, etc. Si los mandamos en el formato incorrecto, la BAPI devuelve error.

✨
La estrategia: tomar una orden de producción que ya existe en el sistema, consultarla con GetDetail, ver cómo SAP tiene guardados esos valores y usar exactamente esos mismos formatos en el Create.
Cómo usarla en SE37
SE37 1. Buscar: BAPI_PRODORD_GETDETAIL
2. En IMPORT → NUMBER = número de orden existente (ej: 000001234567)
3. Ejecutar con F8
4. Revisar EXPORT → ORDER_OBJECTS para ver todos los campos
5. Anotar: MATERIAL, PLANT, ORDER_TYPE, QUANTITY y formato de fechas
💡
Conexión con el código: en un programa ABAP, haríamos exactamente lo mismo — primero llamamos GetDetail para validar que el material existe y tiene el formato correcto, y luego pasamos esos datos al Create. Eso es exactamente lo que hace un desarrollador.

Formato de fecha es crítico. Si la fecha no va en DD.MM.AAAA la BAPI devuelve error inmediatamente en la tabla RETURN. Es el error más común.
Conexión con el código ABAP

Cada uno de estos campos en SE37 corresponde exactamente a un parámetro en el código:

ABAP CALL FUNCTION 'BAPI_PRODORD_CREATE' EXPORTING material = '322B46433R' "← MATERIAL plant = '1000' "← PLANT order_type = 'PP01' "← ORDER_TYPE quantity = 10 "← QUANTITY basic_start_date = '10.04.2026' "← START_DATE basic_end_date = '17.04.2026' "← END_DATE IMPORTING order_number = lv_orden. "← lo que devuelve

Lo que ejecutas manualmente en SE37 es exactamente lo que el código hace automáticamente. El programa no hace nada diferente — solo lo hace sin que tú tengas que escribirlo a mano cada vez.

Paso 4 de 6

Ejecución y resultado
Ejecutar y leer el resultado
Presionamos F8 y SAP nos devuelve una respuesta. Saber leer esa respuesta es tan importante como saber ejecutar la BAPI.
Secuencia de ejecución en SE37 — paso a paso
SE37 · Llamada 1 1. Buscar BAPI_PRODORD_CREATE en SE37
2. Rellenar los campos mínimos en pestaña IMPORT
3. Ejecutar con F8
4. Revisar pestaña TABLES → RETURN
5. Si TYPE = S → anotar el ORDER_NUMBER de pestaña EXPORT
Si TYPE = E → leer MESSAGE y corregir datos. No continuar.
⚠️
Importante: en este punto la orden existe solo en memoria temporal de SAP. Todavía NO está grabada en base de datos. Si cierras la sesión o no ejecutas el COMMIT, desaparece.
SE37 · Llamada 2 — El COMMIT 6. Volver a SE37 y buscar BAPI_TRANSACTION_COMMIT
7. En el campo WAIT introducir X
8. Ejecutar con F8
9. Ahora sí la orden queda grabada definitivamente en SAP
¿Por qué existe el COMMIT? — La analogía del carrito
Imagina que estás comprando en un supermercado online. Vas añadiendo productos al carrito — pero eso no significa que hayas comprado nada. El carrito es temporal. Solo cuando pulsas "Confirmar pedido" la compra se hace real y se descuenta de tu cuenta.

SAP funciona igual. La BAPI añade la orden al "carrito" de la base de datos. El COMMIT es el botón de "Confirmar". Sin él, SAP descarta todo al cerrar la sesión.
— Por eso siempre va en dos llamadas separadas
Interpretar la tabla RETURN

El campo TYPE te dice todo lo que necesitas saber:

S Success
Todo correcto. Ejecutar COMMIT.
E Error
Falló. Ver MESSAGE. No hacer COMMIT.
W Warning
Procesó con advertencia. Evaluar antes de COMMIT.
Cómo lo haría un programa ABAP

El código replica exactamente la misma secuencia de dos llamadas que acabas de hacer manualmente:

ABAP "── Llamada 1: Crear la orden ────────────────── CALL FUNCTION 'BAPI_PRODORD_CREATE' EXPORTING material = lv_material plant = lv_plant ... IMPORTING order_number = lv_orden TABLES return = lt_return. "── Verificar resultado ──────────────────────── READ TABLE lt_return INTO ls_return WITH KEY type = 'E'. IF sy-subrc = 0. "Hubo error → descartar todo CALL FUNCTION 'BAPI_TRANSACTION_ROLLBACK'. ELSE. "── Llamada 2: COMMIT — grabar definitivamente ─ CALL FUNCTION 'BAPI_TRANSACTION_COMMIT' EXPORTING wait = 'X'. "WAIT=X: espera a que grabe antes de continuar WRITE: / 'Orden creada:', lv_orden. ENDIF.
💡
WAIT = X le dice a SAP que espere a que el COMMIT termine completamente antes de seguir ejecutando el programa. Sin eso, el programa podría intentar leer la orden antes de que exista.

Validación final
Verificar la orden en CO03
El paso que conecta todo. Aquí ves que lo que hiciste en SE37 — o lo que haría un programa — tiene un resultado real y concreto en SAP.
CO03 Solo lectura · Sin riesgo
Pasos para verificar
SAP 1. Ir a transacción CO03
2. Introducir el número de orden devuelto por la BAPI
3. Ejecutar con Intro o F8
4. Verificar: material, cantidad, fechas, estado
Lo que acabas de hacer manualmente en SE37, un programa ABAP lo puede ejecutar automáticamente con los mismos campos y la misma lógica — para 1 orden o para 10.000. La diferencia no es qué hace, sino quién lo hace y cuántas veces.
